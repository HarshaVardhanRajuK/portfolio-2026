I used to think my job was to write good code.

Clean architecture. Optimized queries. Proper error handling. Tests that pass. That was success.

Then someone asked me: **"How should we price this?"**

And I realized I had no idea what I was actually building.

---

## The Developer Mindset

For the first year of my career, I lived in a comfortable bubble.

Someone would hand me a feature spec:
- "Build a booking system"
- "Add staff management"
- "Integrate payment gateway"

I'd break it down into tasks:
- Design the schema
- Write the APIs
- Build the frontend
- Deploy

Feature shipped. Ticket closed. Move to the next one.

I was good at this. I could build things that worked. Systems that scaled. Code that other developers could maintain.

But I never asked: **Why are we building this?**

---

## The Moment Everything Changed

I'm currently working on a spa and salon management system. It's my third production system after building a hotel PMS and a banquet booking platform.

By now, I thought I understood the pattern:
1. Understand the requirements
2. Design the system
3. Build it
4. Ship it

Then came the pricing discussion.

The board asked: *"Should we charge per booking? Per user? Subscription? Freemium? Usage-based?"*

I sat there, realizing:
- I had built the entire booking engine
- I had designed the database schema
- I had optimized the queries
- I had handled edge cases

But I had **no framework** for answering: *"How should we charge for this?"*

---

## What I Thought vs What I Learned

### I thought:
- My job = write code
- Product decisions = someone else's problem
- Business model = not my concern

### I learned:
- Code is just the implementation
- Every technical decision affects the business
- You can't build the right thing without understanding the "why"

---

## The Questions I Started Asking

After that meeting, I started seeing my work differently.

**Before**, I'd ask:
- "What's the best way to structure this API?"
- "How do I optimize this query?"
- "What's the right database for this?"

**Now**, I ask:
- "Who is this feature for?"
- "What problem does this solve?"
- "How does this create value?"
- "What happens if we charge for this?"

Same code. Different lens.

---

## The Uncomfortable Realization

Here's what hit me hard:

I had built three production systems serving real users. But I couldn't answer basic questions:
- Why did we choose this pricing model?
- What makes our product valuable?
- How do users actually use what I built?
- What features should we build next?

I was a **code executor**, not a **product builder**.

And that needed to change.

---

## What Changed in My Approach

### 1. I Started Attending Business Meetings

Before: "That's not my job"  
Now: "I need to understand the context"

Sitting in on client calls. Listening to sales conversations. Understanding what customers actually want.

### 2. I Started Questioning Requirements

Before: "Okay, I'll build that"  
Now: "Why do we need this? What's the goal?"

Not to be difficult. But to build the right thing.

### 3. I Started Thinking About Users

Before: "The API works, ship it"  
Now: "How will someone actually use this?"

Walking through user flows. Thinking about edge cases from a user perspective, not just a technical one.

### 4. I Started Learning Business Concepts

Before: Ignored anything non-technical  
Now: Reading about pricing models, unit economics, product strategy

Not to become a PM. But to be a better engineer.

---

## The Spa Pricing Problem

Back to that board meeting.

They wanted to explore:
- Freemium (free basic, paid advanced)
- Usage-based (pay per booking)
- Wallet system (recharge and use)

As a developer, my first thought was: *"That's going to be complex to build."*

But then I started thinking:
- Which model helps users adopt faster?
- Which creates predictable revenue?
- Which aligns our success with theirs?
- How does each model affect the architecture?

Suddenly, it wasn't just a technical problem. It was a **system design problem that touches business, user behavior, and engineering**.

---

## What I'm Learning

I'm not a product expert. I'm still primarily an engineer.

But I'm learning to think beyond the code:

**Technical decisions are product decisions.**

Choosing PostgreSQL over MongoDB? That's a product decision.  
Designing the API structure? That's a product decision.  
Deciding what to cache? That's a product decision.

Every line of code either creates value or doesn't.

---

## The Journey Ahead

I'm documenting this transition because:

1. **I'm still learning** - This isn't a "here's what I know" post. It's a "here's what I'm figuring out" post.

2. **Others might relate** - If you're a developer who's starting to think about product, you're not alone.

3. **It's valuable** - Understanding the business makes you a better engineer. Not because you write different code, but because you write the *right* code.

## Final Thought

A year ago, someone asking me about pricing would have felt like asking a carpenter about interior design.

*"That's not my job."*

Now I realize: if you're building the house, you better understand why people want to live in it.

The code is the same. But the thinking is different.

And that makes all the difference.
